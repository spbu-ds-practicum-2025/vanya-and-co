# План тестирования облачного хранилища (MVP)

## Введение

Данный документ описывает сценарии  тестирования MVP системы «Облачное хранилище». Все тесты предназначены для ручного выполнения.

**Охват MVP**: возможность авторизации в системе, реализация файлового хранилища.

**Предусловия для всех тестов**:
- ((Система полностью развернута и доступна))
- API Gateway доступен по адресу `http://localhost:8080`
- ((В системе существуют тестовые счета с известными идентификаторами))

---

## 1. Тестирование регистрации нового пользователя

### Тест 1.1: Регистрация нового пользователя

**Цель**: проверить возможность регистрации нового пользователя.

**Предусловия**:
- Пользователя с данными, которые будут вводиться, не существует в базе данных зарегистрированных пользователей

**Шаги**:
1. Перейти по адресу `http://localhost:8080`; в открывшемся окне выбрать "Регистрация"

**Ожидаемый результат**:
- Открытие страницы авторизации пользователя; открытие окна регистрации нового пользователя

2. Ввод данных нового пользователя

**Ожидаемый результат**:
- Положительный ответ программы
- Внесение данных пользователя в базу данных
- Авторизация пользователя

### Тест 1.2: Попытка регистрации существующего пользователя

**Цель**: проверить, что система корректно обрабатывает попытку регистрации существующего пользователя.

**Предусловия**:
- Пользователь с данными, которые будут вводиться, существует в базе данных зарегистрированных пользователей

**Шаги**:
1. Перейти по адресу `http://localhost:8080`; в открывшемся окне выбрать "Регистрация"

**Ожидаемый результат**:
- Открытие страницы авторизации пользователя; открытие окна регистрации нового пользователя

2. Ввод данных существующего пользователя

**Ожидаемый результат**:
- Отмена регистрации
- Вывод окна ошибки с сообщением "Пользователь с таким логином существует."
- Предложение входа в существующий аккаунт

## 3. Тестирование работы файлового сервиса

### Тест 3.1: Загрузка файла

**Цель**: проверить работу загрузки файла в облачное хранилище

**Предусловия**:
- Пользователь авторизован, вход в аккаунт выполнен

**Шаги**:
1. Нажать на кнопку "Загрузка файла"

**Ожидаемый результат**:
- Открытие окна с кнопками: "Выбирите файл", "Upload file".

2. Нажать на кнопку "Выбирите файл"

**Ожидаемый результат**:
- Открытие окна для выбора файла пользователем

3. Выбор пользователем загружаемого файла
   
**Ожидаемый результат**:
- Файл корректно выбран
- На странице прописано имя выбранного файла

4. Нажать на кнопку "Upload file"

**Ожидаемый результат**:
- Появление окна с подтверждением загрузки файла

5. Вернуться на страницу назад, нажать на кнопку "Show"

**Ожидаемый результат**:
- Открытие окна со списком загруженных файлов, радом с каждым файлом кнопки "Download" и "Detete"
   

### Тест 3.2: Скачивание файла

**Цель**: проверить корректность работы функции скачивания файла из облачного хранилища

**Предусловия**:
- Пользователь авторизован, выполнен вход в аккаунт
- Открыта страница просмотра скачанных файлов 

**Шаги**:
1. Возле файла нажать на кнопку "Download", проверить, что файл действительно выгружен

**Ожидаемый результат**:
- Выбранный файл корректно выгружен

### Тест 3.3: Удаление файла 

**Цель**: проверить работоспособность функции удаления

**Предусловия**:
- У пользователя есть файлы, загруженные в облачное хранилище
- Выполнен вход в аккаунт 

**Шаги**:
1. Перейти на страницу просмотра скачанных файлов

**Ожидаемый результат**:
- Отображаются все загруженные файлы
- Отображается файл, который нужно удалить из облачного хранилища
- Возле файла есть кнопки "Download" и "Delete"
  
2. Возле нужного файла нажать на кнопку "Delete"
   **Ожидаемый результат**:
   - Из облачного хранилища удален требуемый файл

## 4. Тестирование конкурентности

### Тест 4.1: Множественные параллельные переводы с одного счета

**Цель**: проверить, как система обрабатывает одновременные запросы.

**Предусловия**:
- Счет А существует с балансом 1000.00 RUB
- Существуют счета Б, В, Г

**Шаги**:
1. Одновременно (в течение 1 секунды) отправить несколько HTTP-запросов с разными ключами идемпотентности:
   - Перевод 300.00 RUB со счета А на счет Б (ключ: KEY-A1)
   - Перевод 300.00 RUB со счета А на счет В (ключ: KEY-A2)
   - Перевод 300.00 RUB со счета А на счет Г (ключ: KEY-A3)
   - Перевод 300.00 RUB со счета А на счет Д (ключ: KEY-A4)
   - Перевод 300.00 RUB со счета А на счет Е (ключ: KEY-A5)
2. Дождаться результатов всех операций
3. Проверить балансы всех счетов

**Ожидаемый результат**:
- 3 перевода выполняются успешно (HTTP 200)
- 2 перевода возвращают ошибку "Недостаточно средств" (HTTP 400)
- Итоговый баланс счета А = 100.00 RUB (после 3 успешных переводов по 300.00)
- Баланс НИКОГДА не становится отрицательным
- Сумма всех балансов в системе остается константой
- Нет race conditions или lost updates

## 5. Комплексные сквозные сценарии

### Тест 5.1: Полный жизненный цикл переводов

**Цель**: проверить типичный сценарий использования системы.

**Предусловия**:
- Счет А существует с балансом 1000.00 RUB
- Счет Б существует с балансом 500.00 RUB
- Счет В существует с балансом 0.00 RUB

**Шаги**:
1. Проверить начальный баланс счета А: `GET /accounts/{account-A}`
2. Перевести 100.00 RUB со счета А на счет Б: `POST /accounts/{account-A}/transfers`
3. Перевести 50.00 RUB со счета А на счет В: `POST /accounts/{account-A}/transfers`
4. Проверить баланс счета А: `GET /accounts/{account-A}` (ожидаем 850.00 RUB)
5. Просмотреть историю счета А: `GET /accounts/{account-A}/operations`
6. Проверить баланс счета Б: `GET /accounts/{account-B}` (ожидаем 600.00 RUB)
7. Проверить баланс счета В: `GET /accounts/{account-V}` (ожидаем 50.00 RUB)

**Ожидаемый результат**:
- Все операции выполняются успешно
- Все балансы корректны на каждом этапе
- История операций содержит все выполненные переводы
- Общая сумма всех балансов сохраняется

### Тест 5.2: Цепочка переводов между несколькими пользователями

**Цель**: проверить корректность выполнения последовательных переводов через нескольких участников.

**Предусловия**:
- Счет А существует с балансом 1000.00 RUB
- Счет Б существует с балансом 0.00 RUB
- Счет В существует с балансом 0.00 RUB
- Счет Г существует с балансом 0.00 RUB

**Шаги**:
1. Отправить HTTP-запрос на перевод 500.00 RUB со счета А на счет Б: `POST /accounts/{account-A}/transfers`
2. Отправить HTTP-запрос на перевод 300.00 RUB со счета Б на счет В: `POST /accounts/{account-B}/transfers`
3. Отправить HTTP-запрос на перевод 100.00 RUB со счета В на счет Г: `POST /accounts/{account-V}/transfers`
4. Проверить балансы всех четырех счетов через GET-запросы

**Ожидаемый результат**:
- Все переводы выполнены успешно
- Баланс счета А: 500.00 RUB
- Баланс счета Б: 200.00 RUB (500 - 300)
- Баланс счета В: 200.00 RUB (300 - 100)
- Баланс счета Г: 100.00 RUB
- Общая сумма балансов: 1000.00 RUB (неизменна)
- Все операции отображаются в соответствующих историях

## 6. Проверка консистентности данных

### Тест 6.1: Согласованность данных в истории операций

**Цель**: проверить, что история операций согласуется с текущим состоянием балансов.

**Предусловия**:
- Счет А существует с известным начальным балансом

**Шаги**:
1. Запомнить начальный баланс счета А через `GET /accounts/{account-A}`
2. Выполнить несколько переводов (минимум 3 операции)
3. Запросить полную историю операций: `GET /accounts/{account-A}/operations`
4. Вручную пересчитать баланс на основе истории операций
5. Запросить текущий баланс через `GET /accounts/{account-A}` и сравнить

**Ожидаемый результат**:
- Расчетный баланс (начальный баланс + все операции) совпадает с фактическим
- Все операции учтены в истории
- Нет расхождений
